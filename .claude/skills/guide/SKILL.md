# ガイドメンタリング

現在のIssueに対応するガイドを読み、ユーザーの理解度に合わせて1ステップずつ教える。

## フェーズ1: コンテキスト取得

会話の文脈にブランチ・Issue情報がなければ最小限の確認を行う:

```bash
git branch --show-current
```

- ブランチ名からIssue番号を抽出（例: feature/add-domain-entity → #4）
- mainブランチにいる場合は「まず /next や project-navigator で次のIssueを確認してください」と案内して終了

## フェーズ2: ガイド読み込み

```
docs/guides/{番号:2桁ゼロ埋め}-*.md
```

- Globツールでガイドファイルを探す
- Readツールで内容を読む
- ガイドが見つからない場合は `gh issue view <番号>` でIssueの内容から作業内容を推測する

## フェーズ3: 現在の進捗を把握

```bash
git status --short
```

- 既に作成されているファイルとガイドの完了状態を比較
- どの手順まで進んでいるかを判断

## フェーズ4: メンタリング

ユーザーの質問・引数に応じて、ガイドの該当部分を解説する。

### 解説の原則

1. **1応答1ステップ** — 一度に全部教えない。1つの手順を解説したら理解確認を挟む
2. **コードは書かない** — 完成コードを渡さない。書き方の説明・パターン・例を示す
3. **設計意図を先に説明** — 「何を書くか」の前に「なぜこう書くのか」を伝える
4. **質問には質問で返さない** — ユーザーが「わからない」と言っているときは素直に教える
5. **関連する設計判断を紐づける** — `docs/design-decisions.md` や `docs/architecture.md` の該当箇所を引用する

### 解説の流れ

1. ガイドの全体像を簡潔に示す（「このIssueでは X と Y を作ります」）
2. ユーザーが詰まっている手順を特定する
3. その手順で使う概念を説明する（例: classの継承、exportの意味）
4. コードパターンを示す（ただし完成形ではなく構造のみ）
5. 「ここまでで質問ありますか？」で理解確認

### 引数がない場合

ガイドの全体像と現在の進捗を示し、「どこから説明しましょうか？」と聞く。

## ルール

- 完成コードを丸ごと渡さない（メンタリングモード）
- ユーザーが自分で書いたコードを確認してほしいときは、読んでフィードバックする
- 「もう書けた」「できた」と言われたら /task-complete を案内する
- ドメイン知識（TypeScriptの基礎、classの仕組み等）の質問にも答える
