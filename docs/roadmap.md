# Learning Roadmap

このリポジトリは「何を・どの順で学ぶか」を定める。
「なぜその進め方をするか（予測先行・2つの合格条件・学習ログ）」は
[learning-method.md](learning-method.md) を参照（学習方法の唯一の正典）。

順序の原則：下のフェーズは上を前提にする。飛ばさない。

```
phase1            phase2                  phase3（別リポ）
クリーンアーキ → DDD 戦術パターン      → CQRS / Event Sourcing
層分離・依存逆転   VO→リッチEntity→集約→  events→Outbox→Projector→Read Model
（plain TS）      ドメインイベント       実stack(Effect/Drizzle/SQLite)で縮尺模型
─────────────────────────────────────────────────────────────
壁を1枚ずつ：phase1-2 は「概念」を慣れた書き方(plain TS)で身につける。
phase3 は同じ概念を「案件の書き方(Effect)」で再表現する＝idiom だけ上乗せ。
```

## Phase 1: クリーンアーキテクチャの基礎（Issue #1〜#14）

**目的:** 層分離・依存性逆転・DIを体感する

**学ぶこと:**
- オニオンアーキテクチャの各層（Domain / UseCase / Infra / Presentation）
- 依存方向の制御（外→内。Domain は何にも依存しない）
- Repository インターフェースによる依存性逆転
- Fake Repository による単体テスト
- 手動DIによる配線

**ゴール:** Issue の CRUD が Domain → UseCase → Infra → Presentation の全層を通して動くこと

**ガイド:** [docs/guides/phase1/](guides/phase1/)

---

## Phase 2: DDD 戦術パターン（Issue #23〜#27 ＋ ドメインイベント）

**目的:** Phase 1 のコードをリファクタリングし、DDD の戦術パターンを導入する

**前提:** Phase 1 の層分離・依存逆転は引き続き守る。その上に DDD の要素を付加する。

**学ぶこと（すべて必須）:**

| パターン | 一言で言うと | このリポジトリでは |
|---------|------------|------------------|
| Value Object | 値に意味とルールを持たせる | `IssueTitle`, `IssueId` を導入し、バリデーションをVOに閉じ込める |
| Rich Entity | 状態遷移のルールをオブジェクト自身が守る | `issue.close()`, `issue.reopen()` メソッドで遷移ルールを表現 |
| Aggregate | 整合性を保証する境界を引く | Issue を Aggregate Root にし、Comment を子エンティティとして管理 |
| **ドメインイベント** | 「何が起きたか」を事実として記録する | `IssueClosed` などを集約から発行する |

> **変更点:** ドメインイベントは旧ロードマップで「応用・余裕があれば」だったが、
> Phase 3（Event Sourcing）の前提なので **必須に昇格**した。
> 専用 Issue は Phase 3 着手前に新設する。

**学ぶこと（応用・余裕があれば）:**

| パターン | 一言で言うと |
|---------|------------|
| Domain Service | 単一Entityに属さないドメインロジックの置き場 |

**ゴール:** 各パターンを「なぜ必要か」含めて自分の言葉で説明でき、最小スライスを自分で実装できること
（[2つの合格条件](learning-method.md#2-合格条件2つ測れる形にする)）

**方法:** Phase 1 で作ったコードの「困りごと」を起点に、パターンで解決するリファクタリング

**ガイド:** [docs/guides/phase2/](guides/phase2/)

## Phase 3: CQRS / Event Sourcing（**別リポ**・実 stack 準拠の縮尺模型）

**目的:** 「何が起きたか」の履歴から現在状態を組み立てる設計を、
案件 yasuraku と同じ書き方(idiom)で、自分の手で一度実装する

**なぜ別リポか:** Phase 1-2 で「概念」は plain TS で身につけた。
Phase 3 は同じ概念を**案件の書き方(Effect)で再表現する**段階で、目的が違う。
この道場（plain TS + Prisma）に Effect/Drizzle を混ぜると土台がぶれるため、
専用の新リポジトリ（縮尺模型）を切る。**この repo では実装しない。**

**前提:** Phase 2 のドメインイベントが導入済みであること（概念の足場ができていること）

### 採用スタック（案件 yasuraku に寄せる）

| 採用 | 理由 |
|------|------|
| **Effect-TS** | 案件の最大の壁にして本丸。Schema 検証・Context/Layer(DI)・`Effect<A,E>` を全層で使う書き方を、ここで自作して身につける |
| **Drizzle ORM** | 案件と同じ。Prisma を知っていれば構文差で数時間 |
| **SQLite** | 案件は Turso(SQLite互換)。学習では素の SQLite + Node で動かしてよい（Cloudflare Workers は摩擦が大きいので省略可） |

> **差分の重さは Effect にほぼ一極集中する。**
> Drizzle / SQLite / Workers / monorepo は接触すれば取れる軽量差。
> だから別リポで気合いを入れる対象は **Effect と ES/CQRS の 2 つだけ**でよい。

### 学ぶこと

| 部品 | 一言で言うと | 縮尺模型では |
|------|------------|------------------|
| Event Store | 起きた事実を起きた順に貯める | `events` テーブルにイベントを追記（楽観ロック付き） |
| Outbox | あとで反映すべきイベントを一時的に置く箱 | 未処理イベントを置く |
| Projector | イベントを読んで画面用テーブルを更新する係 | events を読み Read Model を更新する |
| Read Model | 画面が読みやすいよう整えた現在状態 | 一覧・詳細用テーブル |
| **replay** | イベント列から現在状態を導出する | 案件の `derive-status.ts` 相当。ES の心臓部 |
| 手動トリガー | ローカルで反映を手で走らせる仕組み | `process-outbox` 相当のエンドポイント |

**規模:** 1 Aggregate / 3〜5 イベント / 上記部品の薄い 1 本。
ドメインは請求書を丸写しせず別ドメインでよい（抽象化を強制でき転移が深まる）。

**忠実度の方針:** 本物 yasuraku の形に寄せる。
縮尺模型にすれば、後で本物を読むのが楽になるため。
yasuraku 固有の選択と一般的な定石は、ガイド内で区別して注記する。

**ゴール:** Approve のようなコマンドを
Command → Event Store → Projector → Read Model → Query まで自分で 1 本通し、
「上げたのに出ない＝Projector 未処理」のような現象を自分で再現・説明できること

**推奨の締め:** 実装後に本物 yasuraku の同部分
（`event-store.ts` / `invoice.projector.ts` / `derive-status.ts`）を読み、
「自分の実装と本物の違い」を学習ログに 3 点書く（[answer-key 読み](learning-method.md#6-推奨phase3-のあとに答え合わせ読みをする)）

**ガイド:** 別リポ作成時に、そのリポ内へ `docs/guides/` を新設する

## 学習ペースの目安

「いつ終わるか」は1日に取れる時間で決まる。
下は「集中して手が動いた時間」の目安で、中断・復習・詰まりで実際は 1.3〜1.5 倍に膨らむ前提で見る。

### フェーズ別の作業量（集中時間）

| フェーズ | 目安 | 重い所 |
|---------|------|-------|
| Phase 1 残り（#9-14） | 6〜10h | 配管中心。CRUD を全層で1本通す |
| Phase 2（#23-27, #29） | 12〜20h | 概念の本体。各パターン 3〜5h |
| Phase 3 別リポ（Effect+ES） | 25〜40h | ★最重。Effect 習得が長い柱 |
| 合計 | ≈ 45〜70h | 半分以上が Phase 3 |

差分の重さは **Effect にほぼ一極集中**する。
Drizzle / SQLite / Workers / monorepo は接触すれば取れる軽量差なので、
気合いを入れる対象は Effect と ES/CQRS の2つだけでよい。

### 2トラック並走（重い柱を最後に溜めない）

| トラック | 中身 |
|---------|------|
| メイン | 道場の phase を順に進める（概念を自作する） |
| サブ | 週1回、本物 yasuraku を「読んで説明」する（Effect/ES に早めから接触する） |

一番重い Effect を最後の一発勝負にせず、早め＋間隔をあけて浴びる（分散学習）。

### 1日あたりの時間と到達目安

| 1日 | 週の実働 | 到達（目安） |
|-----|---------|-----------|
| 30分 ×5 | 2.5h | 約 5〜7 か月 |
| 1時間 ×5 | 5h | 約 2.5〜3.5 か月 |
| 2時間 ×5 | 8〜10h | 約 1.5〜2 か月 |

> これは見積りであって締め切りではない。実進捗は Issue で追い、
> [2つの合格条件](learning-method.md#2-合格条件2つ測れる形にする)（再説明＋最小実装）を満たしたら次へ進む。
> 週単位の細かい割り当ては固定するとすぐ陳腐化するため、ここには置かない。
