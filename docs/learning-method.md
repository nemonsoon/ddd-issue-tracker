# 学習メソッド（このリポジトリの「なぜ・どう学ぶか」）

このリポジトリは、案件 **yasuraku**（DDD + TDD + CQRS + Event Sourcing）を
**読めて・自分の言葉で説明でき・自分で実装できる**状態になるための、
案件から独立した自己完結の「道場」である。

> このファイルは学習方法の唯一の正典（Single Source of Truth＝ここだけが大元）。
> roadmap / CLAUDE.md / 各 Issue は「なぜそうするか」をここへリンクし、自分では多くを語らない。

---

## 0. このリポジトリの立ち位置

```
目的: 案件(DDD+TDD+CQRS+ES)を、読めて・説明でき・実装できる状態になる
  └ ddd-issue-tracker = 案件から独立した自己完結の道場
     本物 yasuraku は「読み専用の参照（答え合わせ）」。計画の前提にはしない
```

- 案件で本物コードに触れるかは客先都合。**学習計画をそれに依存させない。**
- 「実装まで自分でやる」合格条件（後述）を満たせる場所は、自分が支配できるこの道場だけ。
- だから ES/CQRS も含め、**一度は必ずこのリポジトリで自分の手で実装する。**

## 1. エンジン：予測先行

**原則：上手い解説を読む前に、まず自分の言葉で予測してみる。**

1サイクルは次の5ステップで回す。

| #   | ステップ           | やること                                                                 |
| --- | ------------------ | ------------------------------------------------------------------------ |
| 1   | **予測する**       | 着手前、学習ログに「これは何で、なぜ要る?」を書く（外れてよい）          |
| 2   | **答え合わせ**     | ガイド / 既存の解説 / 本物コードを“手本”として読む                       |
| 3   | **自己説明**       | 実装中、各判断に「なぜこの選択? 他の選択肢とどう違う?」を言葉にする      |
| 4   | **思い出して書く** | ノートを閉じ、学習ログに「再説明」を見ずに書く（予測との差分＝伸びた所） |
| 5   | **実装する**       | テスト→コードでスライスを動かす（コミットが合格の証拠）                  |

### なぜ「読む前に予測」なのか

上手い解説をいきなり読むのは、学習科学的には**弱い学び方**。
読んだだけで「分かった気」になりやすい。
この錯覚を「説明力の錯覚（illusion of explanatory depth＝説明できる気がするだけで実は説明できない現象）」と呼ぶ。

先に自分で予測して外し、答え合わせで直す順序の方が、定着・説明力・実装力が伸びる。
このように「あえて少し難しくする」工夫を「望ましい困難（desirable difficulty）」と呼ぶ。

学校との違いは1点だけ。
**学校は習ってからテスト、ここはテスト（予測）してから習う。**

## 2. 合格条件：2つ（測れる形にする）

ある概念を「理解した」と認める条件は、自己申告ではなく次の2つで測る。

| #   | 合格条件                             | これを外すと落ちる罠                                           |
| --- | ------------------------------------ | -------------------------------------------------------------- |
| ①   | **ノートを見ずに再説明できる**       | 説明だけだと「説明力の錯覚」に落ちる                           |
| ②   | **最小スライスを自分で実装して動く** | 実装だけだと「動くが応用できない（コピペ頼みの理解）」に落ちる |

両方そろって初めて合格。
①は学習ログの「再説明」ブロック、②はコミット＋テストが証拠になる。

### 復習の間隔は「儀式なし」で自然に入る

「数日後にもう一度説明する」という復習の儀式は、運用に乗らないので**採用しない**。
代わりに、学ぶ順序の依存関係が自動で復習を生む。

```
ES を予測するには → ドメインイベントを思い出す必要があり → それには集約を思い出す必要がある
```

新しい概念の「予測」ステップが、前の概念を思い出す練習を兼ねる。
（案件で実際にその層に触れた時も自然な復習になるが、これは当てにせずボーナス扱い。）

## 3. 学ぶ順序（依存関係）

上から1つずつ。下は上を前提にする。

```
VO → リッチな Entity → 集約 → ドメインイベント → events テーブル → Outbox → Projector → Read Model
└──────────── phase2（DDD 戦術パターン） ────────────┘ └─────────── phase3（CQRS / ES） ──────────┘
                             ↑ 旧ロードマップでは「任意」。ES の前提なので【必須】に昇格
```

用語の最小定義（詳しくは phase3 のガイドで都度説明する）。

- **ドメインイベント**: 「何が起きたか」を表す事実の記録（例: `IssueClosed`）。
- **events テーブル**: そのイベントを起きた順に貯める保存先（＝ Event Store）。
- **Outbox**: あとで反映すべきイベントを一時的に置く箱。
- **Projector**: イベントを読んで画面用テーブルを更新する係。
- **Read Model**: 画面が読みやすいよう整えた現在状態のテーブル。

phase3 は本物 yasuraku の形に 1 対 1 で寄せる。
理由は、道場を本物の CQRS / ES の縮尺模型にすれば、後で本物を読むのが楽になるから。
yasuraku 固有の選択と、一般的な定石は、ドキュメント上で区別して注記する。

詳しい順序とゴールは [roadmap.md](roadmap.md) を参照。

## 4. 1サイクルの回し方（＝1 Issue）

各 Issue を「予測先行ループ1周」として扱う。

1. 着手前に学習ログへ **予測** を書く（読む前）。
2. ガイド／手本で **答え合わせ**。
3. テスト先行で **実装**（TDD: RED → GREEN → REFACTOR）。
4. ノートを閉じて **再説明** を見ずに書く。
5. 合格条件2つを満たしたらクローズ。

---

## 5. 学習ログ

Issue ごとに「予測」「再説明」の2ブロックだけ残す。重くしない。
テンプレと使い方は [learning-log/](learning-log/) を参照。

---

## 6. 推奨：phase3 のあとに「答え合わせ読み」をする

予測先行はファイル単位だけでなく、アーキテクチャ全体にも効く。
phase3 で道場の ES を**自分で書いてから**、本物 yasuraku の Event Store / Outbox / Projector を読む。
そして「自分の実装と本物の違い」を学習ログに3点書く。
これは読むだけなので案件都合に左右されず、独立した道場のまま最大の学びを得られる。
